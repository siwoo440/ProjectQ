\# Project Q 개발일지 - Day29



\## 개발 주제



실제 월드 좌표 기반 방 배치 및 카메라 슬라이드 이동 구현



\## 개발 목표



29일차의 목표는 기존 방 이동 방식의 부자연스러움을 개선하는 것이었다.



기존에는 문에 닿으면 현재 방을 삭제하고, 새 방을 같은 위치에 생성한 뒤 플레이어를 EntryPoint로 순간이동시키는 방식이었다.  

기능적으로는 작동했지만, 실제로 옆방으로 이동한다기보다 새로운 방에 소환되는 느낌이 강했다.



이번 작업에서는 아이작의 번제처럼 방이 실제 월드 좌표상에 상하좌우로 배치되고, 문을 통해 이동하면 카메라가 옆방으로 부드럽게 슬라이드되는 구조를 구현하였다.



\## 구현 내용



\### 1. 방 이동 방식 개선 방향 결정



기존 방식은 다음과 같았다.



문에 닿음

→ 현재 방 삭제

→ 새 방 생성

→ 플레이어를 새 방 EntryPoint로 이동



수정 후 목표는 다음과 같다.



방 좌표에 따라 실제 월드 위치에 방 생성

→ 오른쪽 방은 실제 오른쪽에 존재

→ 왼쪽 방은 실제 왼쪽에 존재

→ 위쪽 방은 실제 위쪽에 존재

→ 아래쪽 방은 실제 아래쪽에 존재

→ 문 이동 시 카메라가 옆방으로 부드럽게 이동



이를 통해 방이 새로 소환되는 느낌을 줄이고, 실제로 연결된 옆방으로 이동하는 느낌을 만들었다.



\### 2. RoomData.gridPosition 기반 월드 좌표 변환 적용



RoomData의 gridPosition을 실제 월드 좌표로 변환하는 구조를 추가하였다.



기준 방 크기는 다음과 같이 설정하였다.



\- Room World Size X: 20

\- Room World Size Y: 12



좌표 변환 예시는 다음과 같다.



| RoomData.gridPosition | 실제 월드 위치 |

|---|---|

| (0, 0) | (0, 0, 0) |

| (1, 0) | (20, 0, 0) |

| (-1, 0) | (-20, 0, 0) |

| (0, 1) | (0, 12, 0) |

| (0, -1) | (0, -12, 0) |



이제 방은 항상 같은 위치에 새로 생성되는 것이 아니라, 맵 좌표에 따라 실제 월드 위치에 배치된다.



\### 3. RoomSceneController 구조 변경



RoomSceneController를 기존의 단일 방 생성 방식에서 여러 방 관리 방식으로 수정하였다.



기존 구조는 현재 방 하나만 관리하였다.



수정 후에는 다음 데이터를 관리한다.



\- 생성된 방 오브젝트 Dictionary

\- 생성된 방 컨트롤러 Dictionary

\- 현재 방 컨트롤러

\- 현재 방 주변 방 로드

\- 멀어진 방 정리



이를 통해 이미 생성된 방은 다시 생성하지 않고, 현재 방과 인접 방을 실제 위치에 유지할 수 있게 되었다.



\### 4. 현재 방과 인접 방 미리 생성



방 이동이 자연스럽게 보이도록 현재 방과 상하좌우 인접 방을 미리 생성하도록 수정하였다.



처리 흐름은 다음과 같다.



현재 방 좌표 확인

→ 현재 방 생성

→ 위쪽 방이 있으면 생성

→ 아래쪽 방이 있으면 생성

→ 왼쪽 방이 있으면 생성

→ 오른쪽 방이 있으면 생성



인접한 방이 실제 월드 좌표에 미리 존재하기 때문에, 문 이동 시 방이 갑자기 생성되는 느낌이 줄어들었다.



\### 5. 생성된 방 Dictionary 관리



이미 생성된 방을 중복 생성하지 않도록 Dictionary 구조를 사용하였다.



관리하는 Dictionary는 다음과 같다.



\- spawnedRoomObjects

\- spawnedRoomControllers



spawnedRoomObjects는 방 좌표와 방 GameObject를 저장한다.  

spawnedRoomControllers는 방 좌표와 RoomController를 저장한다.



이를 통해 같은 방에 다시 들어갔을 때 기존 방 데이터를 재사용할 수 있게 되었다.



\### 6. RoomController 초기화 구조 수정



이전에는 RoomController가 RoomMapManager의 현재 방 좌표를 기준으로 문 연결을 확인하였다.



하지만 이제는 인접 방도 미리 생성되기 때문에, 각 방은 자기 자신의 RoomData를 기준으로 문 연결을 확인해야 한다.



따라서 RoomController.Initialize에 RoomData를 전달하도록 수정하였다.



수정 후 문 연결 확인 기준은 다음과 같다.



현재 방의 roomData.gridPosition

→ 문 방향 계산

→ 해당 방향에 RoomData가 있는지 확인

→ 있으면 문 활성화

→ 없으면 문 비활성화



이를 통해 현재 방뿐만 아니라 미리 생성된 인접 방에서도 문 연결 상태가 올바르게 표시되도록 하였다.



\### 7. 전투 시작 시점 분리



인접 방을 미리 생성하더라도, 아직 들어가지 않은 방에서 전투가 시작되면 안 된다.



따라서 RoomController.Initialize에서 전투를 바로 시작하지 않고, 실제로 플레이어가 현재 방에 입장했을 때 EnterRoom을 호출하도록 구조를 분리하였다.



수정 후 흐름은 다음과 같다.



방 프리팹 생성

→ RoomController.Initialize 실행

→ 문 연결 설정

→ 구조물 랜덤 배치

→ 전투는 아직 시작하지 않음

→ 플레이어가 해당 방에 입장

→ RoomController.EnterRoom 실행

→ 전투방이면 RoomBattleController 시작



이 구조 덕분에 인접 방을 미리 생성해도 적이 미리 스폰되지 않는다.



\### 8. 카메라 슬라이드 이동 구현



문 이동 시 카메라가 순간이동하지 않고, 현재 방에서 다음 방으로 부드럽게 이동하도록 구현하였다.



카메라 이동에는 Lerp와 부드러운 보간값을 사용하였다.



설정값은 다음과 같다.



\- Camera Move Duration: 0.35

\- Camera Offset: (0, 0, -10)



동작 흐름은 다음과 같다.



문에 닿음

→ 현재 방 좌표 변경

→ 다음 방 기준 카메라 목표 위치 계산

→ 카메라가 0.35초 동안 목표 위치로 이동

→ 이동 완료 후 최종 위치 보정



이를 통해 방이 실제로 옆에 이어져 있는 것처럼 보이는 연출이 가능해졌다.



\### 9. 플레이어 EntryPoint 이동 유지



방 이동 시 플레이어는 이동한 방향에 따라 다음 방의 반대쪽 EntryPoint로 이동한다.



예시는 다음과 같다.



| 이동 방향 | 배치되는 EntryPoint |

|---|---|

| Up | Entry\_FromBottom |

| Down | Entry\_FromTop |

| Left | Entry\_FromRight |

| Right | Entry\_FromLeft |



이를 통해 오른쪽 방으로 이동하면 플레이어가 오른쪽 방의 왼쪽 입구 쪽에 배치되고, 위쪽 방으로 이동하면 위쪽 방의 아래쪽 입구에 배치된다.



\### 10. 방 이동 중 플레이어 조작 잠금 구현



카메라가 슬라이드되는 동안 플레이어가 움직이면 위치가 꼬이거나 문 Trigger에 다시 닿을 수 있다.



이를 방지하기 위해 방 이동 중 특정 스크립트를 잠시 비활성화하는 기능을 추가하였다.



RoomSceneController에 Behaviours To Disable During Room Transition 배열을 추가하고, 여기에 PlayerController를 연결하였다.



처음에는 Player 오브젝트를 통째로 넣었을 때 CircleCollider2D가 연결되는 문제가 있었다.  

이는 잘못된 설정이므로, Player 오브젝트 자체가 아니라 Inspector에서 PlayerController 컴포넌트 제목 부분을 직접 드래그하여 연결하는 방식으로 해결하였다.



정상 설정 후 방 이동 중에는 PlayerController가 잠시 비활성화되고, 이동이 끝나면 다시 활성화된다.



\### 11. 플레이어 속도 초기화 처리



방 이동 중 플레이어가 기존 이동 속도를 유지하지 않도록 Rigidbody2D의 속도를 0으로 초기화하였다.



이를 통해 카메라 이동 중 플레이어가 미끄러지거나 문 Trigger 안에서 불필요하게 움직이는 문제를 줄였다.



\### 12. 멀어진 방 정리 기능 추가



현재 방과 너무 멀어진 방은 계속 유지할 필요가 없기 때문에, keepRoomDistance 값을 기준으로 멀어진 방을 정리하도록 하였다.



현재 설정은 다음과 같다.



\- Keep Room Distance: 1



이 값은 현재 방과 인접한 방 정도만 유지하고, 그보다 멀어진 방은 삭제하는 기준이다.



이를 통해 방이 계속 누적되어 Hierarchy가 과도하게 늘어나는 문제를 방지하였다.



\### 13. 기존 전투방 기능 유지 확인



방 배치 방식이 변경된 뒤에도 기존 전투방 기능이 정상적으로 작동하는지 확인하였다.



확인한 기능은 다음과 같다.



\- 전투방 입장 시 적 스폰

\- 문 잠금

\- 문 색상 변경

\- DoorBlocker Collider 활성화

\- 적 처치 후 방 클리어

\- 문 잠금 해제

\- RoomData.isCleared 갱신

\- 미니맵 갱신

\- 이미 클리어한 방 재입장 시 전투 반복 방지



\### 14. 기존 방 타입별 프리팹 구조 유지



28일차에 구현한 RoomPrefabDatabase 구조도 그대로 유지하였다.



즉, 방은 실제 월드 좌표에 생성되지만, 어떤 방 프리팹을 사용할지는 여전히 아래 기준을 따른다.



현재 RegionType

→ 현재 RoomType

→ RoomPrefabDatabase에서 후보 프리팹 확인

→ RoomData.roomPrefabIndex 기준으로 프리팹 선택

→ 해당 방을 실제 월드 좌표에 생성



이를 통해 지역별 방 프리팹 관리 구조와 실제 월드 배치 구조가 함께 작동하도록 하였다.

\## 테스트 내용



\### 시작 방 생성 테스트



GameScene 실행 시 시작 방이 RoomData.gridPosition 기준 월드 좌표에 생성되는지 확인하였다.



\### 인접 방 생성 테스트



현재 방 주변의 상하좌우 방이 실제 월드 좌표에 미리 생성되는지 확인하였다.



\### 문 이동 테스트



문에 닿았을 때 다음 방으로 이동하고, 기존처럼 방이 소환되는 느낌이 줄어드는지 확인하였다.



\### 카메라 슬라이드 테스트



문 이동 시 카메라가 다음 방 중심으로 부드럽게 이동하는지 확인하였다.



\### 플레이어 조작 잠금 테스트



방 이동 중 PlayerController가 잠시 비활성화되고, 이동 완료 후 다시 활성화되는지 확인하였다.



\### Inspector 연결 문제 해결



Behaviours To Disable During Room Transition에 Player 오브젝트를 통째로 넣으면 CircleCollider2D가 연결되는 문제가 있었다.



해결 방법은 다음과 같다.



\- Player 오브젝트 전체를 넣지 않음

\- Inspector에서 PlayerController 컴포넌트 제목 부분을 직접 드래그

\- Behaviours To Disable During Room Transition에 PlayerController만 연결



이 방식으로 방 이동 중 조작 잠금이 정상 작동하는 것을 확인하였다.



\### 전투방 테스트



월드 좌표 기반 방 이동으로 변경한 뒤에도 전투방 입장 시 적이 정상적으로 스폰되는지 확인하였다.



\### 문 잠금 테스트



전투 중 문이 잠기고, 방 클리어 후 다시 문이 열리는지 확인하였다.



\### 이벤트방 / 휴식방 테스트



Event 방과 Rest 방에서는 적이 스폰되지 않고, 문 잠금도 발생하지 않는지 확인하였다.



\### 멀어진 방 정리 테스트



현재 방에서 일정 거리 이상 멀어진 방이 자동으로 정리되는지 확인하였다.

이제는 RoomData.gridPosition을 기준으로 방이 실제 월드 좌표에 배치되고, 문 이동 시 카메라가 옆방으로 부드럽게 슬라이드된다.



이를 통해 아이작의 번제처럼 방이 실제로 상하좌우에 이어져 있는 느낌을 줄 수 있는 기반이 마련되었다.

이번 작업에서는 아이작의 번제처럼 방이 실제 월드 좌표상에 상하좌우로 배치되고, 문을 통해 이동하면 카메라가 옆방으로 부드럽게 슬라이드되는 구조를 구현하였다.



\## 구현 내용



\### 1. 방 이동 방식 개선 방향 결정



기존 방식은 다음과 같았다.



문에 닿음

→ 현재 방 삭제

→ 새 방 생성

→ 플레이어를 새 방 EntryPoint로 이동



수정 후 목표는 다음과 같다.



방 좌표에 따라 실제 월드 위치에 방 생성

→ 오른쪽 방은 실제 오른쪽에 존재

→ 왼쪽 방은 실제 왼쪽에 존재

→ 위쪽 방은 실제 위쪽에 존재

→ 아래쪽 방은 실제 아래쪽에 존재

→ 문 이동 시 카메라가 옆방으로 부드럽게 이동



이를 통해 방이 새로 소환되는 느낌을 줄이고, 실제로 연결된 옆방으로 이동하는 느낌을 만들었다.



\### 2. RoomData.gridPosition 기반 월드 좌표 변환 적용



RoomData의 gridPosition을 실제 월드 좌표로 변환하는 구조를 추가하였다.



기준 방 크기는 다음과 같이 설정하였다.



\- Room World Size X: 20

\- Room World Size Y: 12



좌표 변환 예시는 다음과 같다.



| RoomData.gridPosition | 실제 월드 위치 |

|---|---|

| (0, 0) | (0, 0, 0) |

| (1, 0) | (20, 0, 0) |

| (-1, 0) | (-20, 0, 0) |

| (0, 1) | (0, 12, 0) |

| (0, -1) | (0, -12, 0) |



이제 방은 항상 같은 위치에 새로 생성되는 것이 아니라, 맵 좌표에 따라 실제 월드 위치에 배치된다.



\### 3. RoomSceneController 구조 변경



RoomSceneController를 기존의 단일 방 생성 방식에서 여러 방 관리 방식으로 수정하였다.



기존 구조는 현재 방 하나만 관리하였다.



수정 후에는 다음 데이터를 관리한다.



\- 생성된 방 오브젝트 Dictionary

\- 생성된 방 컨트롤러 Dictionary

\- 현재 방 컨트롤러

\- 현재 방 주변 방 로드

\- 멀어진 방 정리



이를 통해 이미 생성된 방은 다시 생성하지 않고, 현재 방과 인접 방을 실제 위치에 유지할 수 있게 되었다.



\### 4. 현재 방과 인접 방 미리 생성



방 이동이 자연스럽게 보이도록 현재 방과 상하좌우 인접 방을 미리 생성하도록 수정하였다.



처리 흐름은 다음과 같다.



현재 방 좌표 확인

→ 현재 방 생성

→ 위쪽 방이 있으면 생성

→ 아래쪽 방이 있으면 생성

→ 왼쪽 방이 있으면 생성

→ 오른쪽 방이 있으면 생성



인접한 방이 실제 월드 좌표에 미리 존재하기 때문에, 문 이동 시 방이 갑자기 생성되는 느낌이 줄어들었다.



\### 5. 생성된 방 Dictionary 관리



이미 생성된 방을 중복 생성하지 않도록 Dictionary 구조를 사용하였다.



관리하는 Dictionary는 다음과 같다.



\- spawnedRoomObjects

\- spawnedRoomControllers



spawnedRoomObjects는 방 좌표와 방 GameObject를 저장한다.  

spawnedRoomControllers는 방 좌표와 RoomController를 저장한다.



이를 통해 같은 방에 다시 들어갔을 때 기존 방 데이터를 재사용할 수 있게 되었다.



\### 6. RoomController 초기화 구조 수정



이전에는 RoomController가 RoomMapManager의 현재 방 좌표를 기준으로 문 연결을 확인하였다.



하지만 이제는 인접 방도 미리 생성되기 때문에, 각 방은 자기 자신의 RoomData를 기준으로 문 연결을 확인해야 한다.



따라서 RoomController.Initialize에 RoomData를 전달하도록 수정하였다.



수정 후 문 연결 확인 기준은 다음과 같다.



현재 방의 roomData.gridPosition

→ 문 방향 계산

→ 해당 방향에 RoomData가 있는지 확인

→ 있으면 문 활성화

→ 없으면 문 비활성화



이를 통해 현재 방뿐만 아니라 미리 생성된 인접 방에서도 문 연결 상태가 올바르게 표시되도록 하였다.



\### 7. 전투 시작 시점 분리



인접 방을 미리 생성하더라도, 아직 들어가지 않은 방에서 전투가 시작되면 안 된다.



따라서 RoomController.Initialize에서 전투를 바로 시작하지 않고, 실제로 플레이어가 현재 방에 입장했을 때 EnterRoom을 호출하도록 구조를 분리하였다.



수정 후 흐름은 다음과 같다.



방 프리팹 생성

→ RoomController.Initialize 실행

→ 문 연결 설정

→ 구조물 랜덤 배치

→ 전투는 아직 시작하지 않음

→ 플레이어가 해당 방에 입장

→ RoomController.EnterRoom 실행

→ 전투방이면 RoomBattleController 시작



이 구조 덕분에 인접 방을 미리 생성해도 적이 미리 스폰되지 않는다.



\### 8. 카메라 슬라이드 이동 구현



문 이동 시 카메라가 순간이동하지 않고, 현재 방에서 다음 방으로 부드럽게 이동하도록 구현하였다.



카메라 이동에는 Lerp와 부드러운 보간값을 사용하였다.



설정값은 다음과 같다.



\- Camera Move Duration: 0.35

\- Camera Offset: (0, 0, -10)



동작 흐름은 다음과 같다.



문에 닿음

→ 현재 방 좌표 변경

→ 다음 방 기준 카메라 목표 위치 계산

→ 카메라가 0.35초 동안 목표 위치로 이동

→ 이동 완료 후 최종 위치 보정



이를 통해 방이 실제로 옆에 이어져 있는 것처럼 보이는 연출이 가능해졌다.



\### 9. 플레이어 EntryPoint 이동 유지



방 이동 시 플레이어는 이동한 방향에 따라 다음 방의 반대쪽 EntryPoint로 이동한다.



예시는 다음과 같다.



| 이동 방향 | 배치되는 EntryPoint |

|---|---|

| Up | Entry\_FromBottom |

| Down | Entry\_FromTop |

| Left | Entry\_FromRight |

| Right | Entry\_FromLeft |



이를 통해 오른쪽 방으로 이동하면 플레이어가 오른쪽 방의 왼쪽 입구 쪽에 배치되고, 위쪽 방으로 이동하면 위쪽 방의 아래쪽 입구에 배치된다.



\### 10. 방 이동 중 플레이어 조작 잠금 구현



카메라가 슬라이드되는 동안 플레이어가 움직이면 위치가 꼬이거나 문 Trigger에 다시 닿을 수 있다.



이를 방지하기 위해 방 이동 중 특정 스크립트를 잠시 비활성화하는 기능을 추가하였다.



RoomSceneController에 Behaviours To Disable During Room Transition 배열을 추가하고, 여기에 PlayerController를 연결하였다.



처음에는 Player 오브젝트를 통째로 넣었을 때 CircleCollider2D가 연결되는 문제가 있었다.  

이는 잘못된 설정이므로, Player 오브젝트 자체가 아니라 Inspector에서 PlayerController 컴포넌트 제목 부분을 직접 드래그하여 연결하는 방식으로 해결하였다.



정상 설정 후 방 이동 중에는 PlayerController가 잠시 비활성화되고, 이동이 끝나면 다시 활성화된다.



\### 11. 플레이어 속도 초기화 처리



방 이동 중 플레이어가 기존 이동 속도를 유지하지 않도록 Rigidbody2D의 속도를 0으로 초기화하였다.



이를 통해 카메라 이동 중 플레이어가 미끄러지거나 문 Trigger 안에서 불필요하게 움직이는 문제를 줄였다.



\### 12. 멀어진 방 정리 기능 추가



현재 방과 너무 멀어진 방은 계속 유지할 필요가 없기 때문에, keepRoomDistance 값을 기준으로 멀어진 방을 정리하도록 하였다.



현재 설정은 다음과 같다.



\- Keep Room Distance: 1



이 값은 현재 방과 인접한 방 정도만 유지하고, 그보다 멀어진 방은 삭제하는 기준이다.



이를 통해 방이 계속 누적되어 Hierarchy가 과도하게 늘어나는 문제를 방지하였다.



\### 13. 기존 전투방 기능 유지 확인



방 배치 방식이 변경된 뒤에도 기존 전투방 기능이 정상적으로 작동하는지 확인하였다.



확인한 기능은 다음과 같다.



\- 전투방 입장 시 적 스폰

\- 문 잠금

\- 문 색상 변경

\- DoorBlocker Collider 활성화

\- 적 처치 후 방 클리어

\- 문 잠금 해제

\- RoomData.isCleared 갱신

\- 미니맵 갱신

\- 이미 클리어한 방 재입장 시 전투 반복 방지



\### 14. 기존 방 타입별 프리팹 구조 유지



28일차에 구현한 RoomPrefabDatabase 구조도 그대로 유지하였다.



즉, 방은 실제 월드 좌표에 생성되지만, 어떤 방 프리팹을 사용할지는 여전히 아래 기준을 따른다.



현재 RegionType

→ 현재 RoomType

→ RoomPrefabDatabase에서 후보 프리팹 확인

→ RoomData.roomPrefabIndex 기준으로 프리팹 선택

→ 해당 방을 실제 월드 좌표에 생성



이를 통해 지역별 방 프리팹 관리 구조와 실제 월드 배치 구조가 함께 작동하도록 하였다.

\## 테스트 내용



\### 시작 방 생성 테스트



GameScene 실행 시 시작 방이 RoomData.gridPosition 기준 월드 좌표에 생성되는지 확인하였다.



\### 인접 방 생성 테스트



현재 방 주변의 상하좌우 방이 실제 월드 좌표에 미리 생성되는지 확인하였다.



\### 문 이동 테스트



문에 닿았을 때 다음 방으로 이동하고, 기존처럼 방이 소환되는 느낌이 줄어드는지 확인하였다.



\### 카메라 슬라이드 테스트



문 이동 시 카메라가 다음 방 중심으로 부드럽게 이동하는지 확인하였다.



\### 플레이어 조작 잠금 테스트



방 이동 중 PlayerController가 잠시 비활성화되고, 이동 완료 후 다시 활성화되는지 확인하였다.



\### Inspector 연결 문제 해결



Behaviours To Disable During Room Transition에 Player 오브젝트를 통째로 넣으면 CircleCollider2D가 연결되는 문제가 있었다.



해결 방법은 다음과 같다.



\- Player 오브젝트 전체를 넣지 않음

\- Inspector에서 PlayerController 컴포넌트 제목 부분을 직접 드래그

\- Behaviours To Disable During Room Transition에 PlayerController만 연결



이 방식으로 방 이동 중 조작 잠금이 정상 작동하는 것을 확인하였다.



\### 전투방 테스트



월드 좌표 기반 방 이동으로 변경한 뒤에도 전투방 입장 시 적이 정상적으로 스폰되는지 확인하였다.



\### 문 잠금 테스트



전투 중 문이 잠기고, 방 클리어 후 다시 문이 열리는지 확인하였다.



\### 이벤트방 / 휴식방 테스트



Event 방과 Rest 방에서는 적이 스폰되지 않고, 문 잠금도 발생하지 않는지 확인하였다.



\### 멀어진 방 정리 테스트



현재 방에서 일정 거리 이상 멀어진 방이 자동으로 정리되는지 확인하였다.



\## 완료 결과



29일차 작업을 통해 Project Q의 방 이동 방식이 크게 개선되었다.



기존에는 방 이동 시 현재 방을 삭제하고 새 방을 같은 위치에 생성하는 방식이었기 때문에, 플레이어가 옆방으로 이동한다기보다 새 방에 소환되는 느낌이 강했다.



이제는 RoomData.gridPosition을 기준으로 방이 실제 월드 좌표에 배치되고, 문 이동 시 카메라가 옆방으로 부드럽게 슬라이드된다.



이를 통해 아이작의 번제처럼 방이 실제로 상하좌우에 이어져 있는 느낌을 줄 수 있는 기반이 마련되었다.



또한 인접 방 미리 생성, 방 Dictionary 관리, 플레이어 조작 잠금, 멀어진 방 정리까지 추가하여 이후 보스방, 포탈, 다음층 이동 구조를 구현하기 좋은 상태가 되었다.
