\# Project Q 개발일지 - Day23



\## 개발 주제



아이작식 랜덤 방 맵 데이터 생성 및 미니맵 UI 구현



\## 개발 목표



23일차의 목표는 기존 Slay the Spire식 노드맵 구조에서 벗어나, 아이작의 번제처럼 방 단위로 탐험하는 랜덤 맵 시스템의 기반을 만드는 것이었다.



이번 단계에서는 실제 방 프리팹 로드, 문 이동, 전투 연결까지 구현하지 않고, 먼저 층마다 랜덤한 방 배치가 생성되고 이를 미니맵으로 표시하는 구조를 구현하였다.



최종적으로는 GameScene 안에서 플레이어가 시작 방에서 출발하고, 상하좌우로 연결된 방을 탐험하며, 끝방에 도달해 보스 또는 정예 전투를 진행하는 구조로 확장할 예정이다.



\## 구현 내용



\### 1. RoomSystem 폴더 구조 생성



새로운 방 탐험 시스템을 기존 MapScene 노드 구조와 분리하기 위해 RoomSystem 전용 폴더를 생성하였다.



생성한 폴더는 다음과 같다.



\- Assets/Scripts/RoomSystem

\- Assets/Scripts/RoomSystem/Data

\- Assets/Scripts/RoomSystem/UI



기존 MapNodeData, MapSceneUI, MapNodeUI는 바로 삭제하지 않고, 새 구조가 안정화된 뒤 정리하기로 하였다.



\### 2. 방 타입 데이터 생성



방의 종류를 구분하기 위해 RoomType enum을 추가하였다.



방 타입은 다음과 같이 구성하였다.



\- Start

\- NormalBattle

\- EliteBattle

\- Event

\- Rest

\- MidBoss

\- FinalBoss



이를 통해 각 방이 시작 방인지, 일반 전투방인지, 정예 전투방인지, 보스방인지 구분할 수 있게 되었다.



\### 3. 층 타입 데이터 생성



층의 구조를 구분하기 위해 FloorType enum을 추가하였다.



층 타입은 다음과 같다.



\- Normal

\- MidBoss

\- FinalBoss



층 타입에 따라 생성되는 방 개수와 끝방의 역할이 달라지도록 설계하였다.



\### 4. 지역 타입 데이터 생성



총 15층 구조를 지역별로 관리하기 위해 RegionType enum을 추가하였다.



지역은 다음과 같이 구성하였다.



\- Forest

\- Beach

\- Desert

\- Snowfield

\- ServerRoom



각 지역은 3개 층으로 구성되며, 전체 구조는 다음과 같다.



\- Forest 1, 2, 3

\- Beach 1, 2, 3

\- Desert 1, 2, 3

\- Snowfield 1, 2, 3

\- ServerRoom 1, 2, 3



\### 5. RoomData 생성



방 하나의 정보를 저장하기 위한 RoomData 클래스를 추가하였다.



RoomData는 다음 정보를 가진다.



\- 방의 격자 좌표

\- 방 타입

\- 방문 여부

\- 클리어 여부

\- 시작 방으로부터의 거리



또한 전투방인지, 보스방인지 확인할 수 있는 함수도 함께 추가하였다.



이를 통해 미니맵 표시, 방 이동, 방 클리어 처리, 보스방 판정의 기반을 마련하였다.



\### 6. FloorData 생성



현재 층 정보를 저장하기 위한 FloorData 클래스를 추가하였다.



FloorData는 다음 정보를 가진다.



\- 현재 층 번호

\- 현재 지역

\- 현재 층 타입

\- 목표 방 개수



층 번호를 기준으로 지역과 층 타입을 자동으로 계산하도록 구성하였다.



예를 들어 1층은 Forest 1이며 일반층이고, 2층은 Forest 2이며 중간보스층, 3층은 Forest 3이며 최종보스층으로 처리된다.



\### 7. RoomMapManager 구현



랜덤 방 맵 생성을 담당하는 RoomMapManager를 추가하였다.



RoomMapManager의 주요 기능은 다음과 같다.



\- 현재 층 번호 기준으로 층 데이터 생성

\- 층 타입에 따라 방 개수 랜덤 결정

\- 시작 방 생성

\- 상하좌우 방향으로 연결된 랜덤 방 배치 생성

\- 시작 방에서 가장 먼 방 계산

\- 가장 먼 방을 끝방으로 지정

\- 방 타입 비율에 따라 일반 방 타입 배정

\- 현재 방 위치 저장

\- 테스트용 방향키 방 이동

\- 미니맵 갱신



층별 방 개수는 다음 규칙을 적용하였다.



| 층 타입 | 방 개수 |

|---|---:|

| 일반층 | 8\~10개 |

| 중간보스층 | 10\~15개 |

| 최종보스층 | 12\~18개 |



\### 8. 끝방 지정 규칙 구현



랜덤으로 생성된 방 중 시작 방에서 가장 먼 방을 끝방으로 지정하였다.



끝방의 타입은 층 타입에 따라 다르게 설정하였다.



| 층 타입 | 끝방 타입 |

|---|---|

| 일반층 | 정예 전투방 |

| 중간보스층 | 중간보스방 |

| 최종보스층 | 최종보스방 |



이 구조를 통해 모든 층에 하나의 목표 방이 생기도록 구성하였다.



\### 9. 방 타입 비율 적용



끝방과 시작 방을 제외한 나머지 방에는 추천 비율을 적용하였다.



비율은 다음과 같다.



| 방 타입 | 비율 |

|---|---:|

| 일반 전투방 | 60% |

| 정예 전투방 | 20% |

| 이벤트방 | 10% |

| 휴식방 | 10% |



아직 이벤트방과 휴식방의 실제 기능은 연결하지 않았지만, 방 데이터상으로는 미리 배정되도록 구성하였다.



\### 10. RoomMinimapCellUI 구현



미니맵에서 방 하나를 표시하기 위한 RoomMinimapCellUI를 추가하였다.



각 방 칸은 방 상태에 따라 다른 표시를 가진다.



\- 현재 방: ●

\- 시작 방: S

\- 미탐사 방: ?

\- 정예 방: E

\- 휴식 방: R

\- 보스 방: B

\- 일반 클리어 방: 빈칸 또는 회색 칸



또한 현재 방, 클리어 방, 미탐사 방, 시작 방, 보스방의 색상을 다르게 적용할 수 있도록 구성하였다.



\### 11. RoomMinimapUI 구현



미니맵 전체 표시를 담당하는 RoomMinimapUI를 추가하였다.



RoomMinimapUI는 RoomMapManager의 방 데이터를 받아와 미니맵 칸을 생성한다.



주요 기능은 다음과 같다.



\- 기존 미니맵 칸 제거

\- 보이는 방 목록 가져오기

\- 방 좌표에 따라 미니맵 칸 생성

\- 현재 층 정보 표시

\- 현재 방 기준으로 미니맵 갱신



\### 12. GameScene에 Room System 오브젝트 추가



GameScene에 새로운 Room System 오브젝트를 추가하였다.



구성은 다음과 같다.



\- ===Room System===

&#x20; - RoomMapManager

&#x20; - RoomMinimapUI



또한 미니맵 표시를 위해 Canvas\_RoomHUD를 추가하였다.



구성은 다음과 같다.



\- Canvas\_RoomHUD

&#x20; - Panel\_Minimap

&#x20;   - Text\_FloorInfo

&#x20;   - CellParent



Panel\_Minimap은 화면 오른쪽 위에 배치하여 현재 층과 방 배치를 확인할 수 있도록 하였다.



\### 13. RoomCellTemplate 프리팹 생성



미니맵 방 칸으로 사용할 RoomCellTemplate 프리팹을 생성하였다.



구성은 다음과 같다.



\- RoomCellTemplate

&#x20; - Text\_Mark



RoomCellTemplate에는 RoomMinimapCellUI 스크립트를 연결하고, Background Image와 Mark Text를 연결하였다.



처음에는 RoomCellTemplate을 비활성화한 상태로 프리팹에 저장하여 생성된 칸이 보이지 않는 문제가 있었지만, 프리팹 자체를 활성화하고 Hierarchy의 템플릿만 비활성화하는 방식으로 수정하여 정상 작동하도록 해결하였다.



\### 14. 미니맵 셀 생성 오류 수정



미니맵의 층 정보 텍스트는 표시되었지만 방 칸이 보이지 않는 문제가 발생하였다.



원인은 다음 중 하나였다.



\- RoomCellTemplate 프리팹이 비활성화된 상태로 저장됨

\- 비활성화된 Hierarchy 오브젝트가 Cell Prefab에 연결됨



수정 후에는 Project 창의 RoomCellTemplate 프리팹을 활성화 상태로 유지하고, RoomMinimapUI의 Cell Prefab에는 Project 창의 프리팹 파일을 연결하였다.



또한 생성된 미니맵 칸이 확실히 보이도록 Instantiate 이후 SetActive(true)를 적용하였다.



\### 15. 현재 방 중심 미니맵 이동 방식 구현



처음에는 미니맵 전체가 고정된 상태에서 현재 방 표시만 이동하였다.



이 방식은 플레이어가 맵을 이동할 때 미니맵의 중심이 바뀌지 않아 아이작식 미니맵 느낌이 부족하였다.



이를 수정하여 현재 방이 항상 미니맵 중앙에 고정되고, 주변 방들이 현재 방 기준으로 상대적으로 움직이도록 변경하였다.



수정 방식은 다음과 같다.



\- RoomMinimapUI에 centerOnCurrentRoom 옵션 추가

\- 방의 실제 좌표에서 현재 방 좌표를 뺀 상대 좌표로 UI 위치 계산

\- 현재 방은 항상 (0, 0)에 표시

\- 이동 시 주변 방들이 반대 방향으로 움직이는 형태 구현



이를 통해 미니맵이 플레이어 중심으로 움직이는 방식에 가까워졌다.



\### 16. 테스트용 방향키 이동 구현



아직 실제 방 문 이동이 구현되지 않았기 때문에, 테스트를 위해 방향키와 WASD로 현재 방을 이동할 수 있도록 하였다.



입력 방식은 다음과 같다.



\- W 또는 위쪽 방향키: 위 방 이동

\- S 또는 아래쪽 방향키: 아래 방 이동

\- A 또는 왼쪽 방향키: 왼쪽 방 이동

\- D 또는 오른쪽 방향키: 오른쪽 방 이동



이동 가능한 방이 있으면 현재 방이 변경되고, 방문 및 클리어 처리 후 미니맵이 갱신된다.

\## 테스트 내용



\### 랜덤 방 생성 테스트



GameScene 실행 시 현재 층 번호에 따라 랜덤한 방 배치가 생성되는지 확인하였다.



\### 층별 방 개수 테스트



RoomMapManager의 Current Floor Number 값을 변경하면서 층별 방 개수가 정상적으로 결정되는지 확인하였다.



확인한 기준은 다음과 같다.



\- 1층: Forest 1, 8\~10개 방

\- 2층: Forest 2, 10\~15개 방

\- 3층: Forest 3, 12\~18개 방

\- 15층: Server Room 3, 12\~18개 방



\### 끝방 지정 테스트



시작 방에서 가장 먼 방이 끝방으로 지정되는지 확인하였다.



일반층에서는 끝방이 정예방으로 지정되고, 중간보스층에서는 중간보스방, 최종보스층에서는 최종보스방으로 지정되는 것을 확인하였다.



\### 미니맵 표시 테스트



미니맵에 현재 방, 미탐사 방, 클리어 방이 표시되는지 확인하였다.



처음에는 방 칸이 보이지 않는 문제가 있었으나, RoomCellTemplate 프리팹 활성화 문제를 수정하여 정상적으로 표시되도록 해결하였다.



\### 현재 방 중심 미니맵 테스트



방을 이동했을 때 현재 방이 미니맵 중앙에 고정되고, 주변 방들이 현재 방 기준으로 움직이는지 확인하였다.



\### 방향키 이동 테스트



WASD와 방향키 입력을 통해 인접한 방으로 이동할 수 있는지 테스트하였다.



연결된 방이 없을 경우 Console에 이동 불가 로그가 출력되는 것을 확인하였다.



\## 완료 결과



23일차 작업을 통해 Project Q에 아이작식 랜덤 방 탐험 구조의 기반이 추가되었다.



이제 층이 시작되면 방 개수가 랜덤으로 결정되고, 시작 방을 기준으로 연결된 랜덤 방 배치가 생성된다.



또한 가장 먼 방은 층 타입에 따라 정예방, 중간보스방, 최종보스방으로 지정되며, 미니맵 UI를 통해 현재 방과 주변 방 상태를 확인할 수 있다.



이번 작업은 실제 전투 연결이나 문 이동까지 구현한 단계는 아니지만, 이후 방 프리팹 로드, 문 이동, 전투방 베리어, 보스방 포탈 구조를 붙이기 위한 핵심 기반이 되었다.
